# PLOD — план реализации и подготовки к тестированию

Актуализировано: 16 июля 2026. Этот документ заменяет greenfield-план: он исходит из фактического состояния репозитория и production-проекта Supabase `tanz-karta`.

## 1. Цель продукта и граница первого теста

PLOD помогает танцору найти релевантное событие в выбранном городе, понять программу и расписание, выбрать конкретную сессию и записаться. Главная проверяемая гипотеза первой закрытой беты: пользователь без объяснений проходит путь `вход/гость → город → поиск/карта → preview-досье → выбор сессии → запись → Мои записи`.

Первый тестовый релиз — web beta на Vercel. Нативная beta следует после стабилизации основного сценария. В первую beta не входят площадки, чат, отзывы, push-уведомления, создание событий пользователями и полноценная админ-панель: они не нужны для проверки основного discovery/booking-сценария.

## 3. Уточнённые продуктовые решения

1. **Одна запись на событие, одна выбранная сессия.** Текущий `unique(user_id, event_id)` сохраняем для MVP; `bookings.session_id` указывает выбранный слот. Если позже понадобится посещать несколько дней одного события отдельно, мигрируем на `unique(user_id, session_id)` и переносим вместимость в `event_sessions`.
2. **Вместимость пока на уровне события.** `events.seats_total/seats_taken` остаются едиными для всего события. Посессионную вместимость не добавляем до подтверждения потребности организаторами.
3. **Отмена за 24 часа считается от забронированной сессии**, а не от первой сессии события. Бесплатное событие подчиняется тому же правилу.
4. **Web beta — основной канал первого теста.** Карта и телефонный OTP не блокируют первую beta. iOS/Android beta начинается после прохождения web release gate.
5. **Организаторов назначает админ.** Самостоятельная заявка на роль — после первой beta.
6. **Отзывы только после `attended`.** Автоматический `attended` допустим для MVP, но в интерфейсе отзыв должен явно относиться к событию и выбранной сессии.
7. **Площадки — отдельный продуктовый контур.** Не возвращаем таб до готовности списка, карточки и хотя бы одного полезного действия.
8. **Офлайн в MVP — только данные.** Не кешируем Google Maps tiles. Persisted query cache подключаем перед нативной beta, если это подтверждается тестами.

## 4. Технические корректировки исходного спека

- Репозиторий использует `src/app`, а не корневой `app`.
- `book_event` уже принимает `(p_event_id, p_session_id)`; старый пример с одним аргументом больше не использовать.
- `cancel_booking` должен получить правило 24 часов и проверять `bookings.session_id`.
- Прямые `insert/update/delete` в `bookings` уже отозваны; все изменения идут через RPC.
- `react-native-mmkv`, Expo Notifications и Edge Functions отсутствуют в зависимостях/репозитории; не считать их частью готового стека.
- Production использует Postgres 17.6; старые замечания про Postgres 14 неактуальны.
- Новые таблицы нужно не только защитить RLS, но и явно предоставить Data API grants согласно настройкам Supabase 2026.
- SECURITY DEFINER helper-функции нельзя оставлять вызываемыми через публичный RPC. Для `book_event/cancel_booking` доступ intentional; для `is_admin/is_organizer/can_see_event/can_chat` — закрыть `EXECUTE` для `public`, `anon`, `authenticated`, не ломая использование внутри RLS.

## 5. План реализации

### R0 — Release foundation и безопасность (P0)

Цель: получить воспроизводимый и безопасный baseline перед новыми фичами.

- Завершить Auth URL Configuration: production Site URL и redirect allowlist.
- Сверить local migrations с production migration history; корректно зарегистрировать/применить booking migration без повторного изменения данных.
- Добавить новую миграцию с revoke для helper SECURITY DEFINER functions.
- Разобрать security advisor: leaked-password protection, grants, RLS helper exposure; зафиксировать intentional warnings для booking RPC.
- Добавить `.env.example` с фактически нужными переменными без секретов.
- Подключить ESLint-конфигурацию и оставить команды `typecheck`, `lint`, `build:web` обязательными.
- Добавить минимальный CI на pull request: install → typecheck → lint → web build.
- Добавить error/empty/loading states для auth и booking; локализовать технические сообщения Supabase.

**Release gate:** чистая сборка и CI; production schema совпадает с репозиторием; нет security warnings, кроме документированных RPC; подтверждение email возвращает в PLOD.

### R1 — Закрытая web beta: discovery → booking (P0)

Цель: дать 5–10 тестировщикам законченный основной сценарий.

- Проверить новый аккаунт: signup → письмо → подтверждение → онбординг → список.
- Реализовать правило отмены за 24 часа в `cancel_booking`; добавить понятное состояние `TOO_LATE`.
- Привязать отображение записи к `booking.session_id`, а не вычислять дату по всем сессиям события.
- Проверить атомарность: два параллельных запроса на последнее место, повторное бронирование, отмена и повторная запись.
- Добавить в досье явные: адрес выбранной сессии, оставшиеся места, состояние CTA, правила отмены.
- ✅ Подготовлены 20 правдоподобных active-событий в 5 городах: бесплатные, чемпионаты и мультисессионные красные нити.
- Добавить простой канал обратной связи: ссылка/кнопка «Сообщить о проблеме» с контекстом route и event id.
- Добавить privacy/terms минимум в виде доступных web-страниц перед внешним приглашением.
- Провести ручной E2E на мобильной ширине Safari/Chrome и проверить accessibility основных интерактивных элементов.

**Release gate:** два независимых тестовых аккаунта проходят весь flow; booking/cancel подтверждены в БД; нет P0/P1 ошибок; заполнена тестовая анкета и назначен канал поддержки.

### R2 — Организаторский контур (P1)

Цель: перестать наполнять базу вручную.

- Создать `event/create`: основные данные → расписание 1..5 сессий → карта/адрес → preview → submit.
- Подключить Storage bucket для фото с ограничением MIME/размера и ownership policies.
- Копировать `card_color/marker_icon` из профиля при создании события.
- Добавить экран «Мои события» со статусами draft/pending/active/rejected/cancelled.
- Реализовать admin moderation MVP: очередь, approve/reject с причиной, аудит действий.
- Добавить RPC/транзакцию публикации, валидирующую сессии, цену, вместимость и роль.
- Обработать отмену события: отмена активных booking и запись notification rows.

**Release gate:** обычный пользователь создаёт pending-событие; админ публикует; организатор публикует согласно роли; событие появляется в discovery без ручного SQL.

### R3 — Нативная beta: карта и устройство (P1)

Цель: выпустить TestFlight/Internal testing с основной ценностью карты.

- Настроить EAS project, bundle identifiers/package name, signing и `eas.json`.
- Ограничить Google Maps keys по bundle id/package и API; не хранить ключи в git.
- Проверить карту, кластеризацию/производительность, golden marker и red-thread polyline на iOS/Android.
- Настроить deep links `plod://` и universal/app links для email confirmation и event sharing.
- Добавить permissions copy и поведение без геолокации.
- Подключить crash reporting и release metadata.
- Решить по результатам тестов, нужен ли persisted query cache; если да — выбрать поддерживаемое Expo-хранилище и кешировать только данные.

**Release gate:** signed builds устанавливаются на реальные устройства; deep links и auth работают; карта не падает без permissions/network; crash reports привязаны к версии.

### R4 — Удержание: уведомления, отзывы, чат (P2)

- Добавить `push_tokens` с привязкой к user/device и безопасной ротацией.
- Реализовать booking confirmation и reminder за 24 часа с идемпотентностью доставки.
- Добавить `finalize_events`: finished + attended; учитывать выбранную сессию и повторный безопасный запуск.
- Реализовать отзыв после attended и агрегированный рейтинг.
- Реализовать event chat через Realtime, pagination, rate limit/reporting и закрытие после события.
- Добавить in-app notification center; push — транспорт, таблица notifications — источник истории.

**Release gate:** cron/функции идемпотентны; повторный запуск не дублирует push; RLS не раскрывает чужие bookings/messages/tokens.

### R5 — Площадки и расширение продукта (P2)

- Список и карточка площадок, поиск по названию/метро, полезное действие «связаться/запросить аренду».
- Создание и модерация площадки, Storage policies, ownership.
- Определить модель занятости: ручной статус заменить расписанием только после подтверждения спроса.
- Вернуть таб «Площадки» после наполнения минимум 10 реальными объектами.
- Социальный шаринг, календарь, подписки на направления/хореографов.

## 6. Сквозная стратегия тестирования

### Автоматически на каждый PR

- TypeScript typecheck.
- ESLint.
- Expo static web build.
- SQL lint/advisors для миграций.
- Тесты чистых функций: фильтры, поиск, выбор ближайшей сессии, форматирование дат.

### Интеграционно для каждого релиза

- RLS matrix: anon/user/organizer/admin для каждой затронутой таблицы.
- RPC concurrency и idempotency.
- Auth redirect для production, preview и native schemes.
- Browser flow на новом storage profile, а не на уже авторизованной сессии.
- Проверка production deployment и Supabase logs после релиза.

### Минимальная тестовая матрица

| Сценарий | Гость | User | Organizer | Admin |
|---|---:|---:|---:|---:|
| Смотреть active events | ✓ | ✓ | ✓ | ✓ |
| Записаться/отменить | — | ✓ | ✓ | ✓ |
| Создать событие | — | pending | active | active |
| Видеть свои pending | — | ✓ | ✓ | ✓ |
| Видеть чужие pending | — | — | — | ✓ |
| Модерировать | — | — | — | ✓ |
| Читать event chat | — | booked only | own event | ✓ |

## 8. Метрики первой beta

- Доля тестировщиков, открывших хотя бы одно досье.
- Доля открывших досье, которые выбрали сессию и нажали booking CTA.
- Успешные booking / попытки booking.
- Доля пользователей, нашедших запись в «Моих записях» без подсказки.
- Ошибки auth и booking на пользователя.
- Время от первого открытия до первого досье и первой записи.
- Качественная обратная связь: доверие к данным, понятность дат/мест, ценность карты против списка.

## 9. Definition of Done для любой фичи

- Есть loading, empty, error и success states.
- Проверены guest/authenticated права и RLS.
- Изменения схемы находятся в migration history и в репозитории.
- TypeScript types обновлены при изменении схемы.
- Нет секретов и service-role ключей в клиенте/репозитории.
- Пройдены typecheck, lint, релевантные тесты и production build.
- Фича проверена на production-like данных и мобильном viewport.
- Для внешнего изменения есть понятный текст ошибки и способ восстановления.
- Документация статуса обновлена; незавершённая фича не показывается в основной навигации.
